# 小米企业赛题五 - SmartFridge 后续开发计划

## 1. 文档目的

这份文档不是泛泛而谈的“想法清单”，而是给当前项目做持续开发的执行手册。目标有 3 个：

1. 把赛题要求拆成可以逐步落地的功能路线。
2. 让你作为小白也能知道每一步先做什么、后做什么、改哪些文件、怎么验收。
3. 给每个阶段配一套可以直接复制给 AI 的 Prompt，尽量降低你写 Prompt 的门槛。

当前项目已经具备一个可运行原型：

- 开发板端已经打通门磁、灯带、AI 推理、LCD 触摸 UI。
- 服务端已经有 Flask、库存/事件接口、Web 页面、视频代理。
- 现在最缺的不是“能不能跑起来”，而是“如何对齐赛题评分点，把原型补成一个稳定、可解释、可测试、可答辩的系统”。

## 2. 赛题要求与当前进度差距分析

| 赛题要求 | 当前进度 | 差距判断 | 优先级 |
| --- | --- | --- | --- |
| 真实事件识别：区分真实放入/取出与手部晃动、短时遮挡、整理无变化 | 已有开门前后对比逻辑 | 还不够稳，容易把无效动作当事件或把复杂操作合并得过粗 | P0 |
| 分层识别：至少做到“是否有新增/减少物体”+“粗粒度分类” | 已有 YOLO 检测与具体名称映射 | 缺少系统化的粗分类层，结果解释能力不够 | P0 |
| 复杂条件适应：低照度、遮挡、反光、堆叠、包装干扰 | 已有补光 + 现有模型 | 需要从策略层和测试层证明系统适应性 | P1 |
| 库存数据库与更新机制 | 已实现库存、事件、状态 | 还缺少更清晰的库存策略、事件状态、冲突修正机制 | P0 |
| 部分取出处理 | 基本没有完整策略 | 必须补，且要可解释、可验证 | P0 |
| 系统闭环与交互：感知-识别-记录-更新-展示-提醒 | 已基本成形 | 还缺少“确认/修正/提醒”闭环 | P1 |
| 端侧部署与性能说明 | 已在 Luckfox Pico Ultra W 跑通 | 需要整理性能、资源占用、部署说明 | P1 |
| 测试评估报告 | 目前只有零散验证 | 需要专门建设测试流程、测试数据、结果记录表 | P0 |
| 扩展能力：多目标同时进出、包装标签辅助、到期提醒 | 有思路，无完整实现 | 适合作为加分项在基础功能稳定后推进 | P2 |

结论：

- 你现在已经不是“从零开始”，而是已经完成了一个比赛原型的 40% 到 55%。
- 下一步最重要的是先补“保底得分项”，也就是评分标准里 20 分、20 分、20 分那几项：真实事件识别、分层识别、库存更新机制。
- 加分项不要一开始就冲。应该先把主链路变得稳定、可解释、可测试，再去做 OCR、到期提醒、多目标同时操作这类扩展功能。

## 3. 开发总原则

后面所有功能都必须遵守下面这些原则，否则会越改越乱：

### 3.1 以实际运行主线为准

当前项目真正运行的主线文件是：

- `deploy/fridge_ai`
- `deploy/fridge_mgr.py`
- `deploy/lcd_ui.py`
- `server/app.py`
- `server/templates/index.html`

`board/` 目录更多是参考代码、早期设计或交叉编译工程，不要一上来就把主要精力放在 `board/` 上。

### 3.2 每个功能都要兼容前面功能

后续功能不能孤立开发，必须遵循这个依赖顺序：

1. 先稳定事件识别。
2. 再在事件识别上叠加粗分类。
3. 再在粗分类结果上做库存更新和部分取出。
4. 再在库存模型上做用户确认、修正、提醒。
5. 最后再做包装标签、OCR、到期提醒、多目标加分项。

### 3.3 先做“可解释”，再做“更聪明”

比赛评委更看重：

- 你的系统为什么这么判断。
- 判断错了怎么修正。
- 你的库存为什么这么更新。
- 测试怎么证明方案有效。

所以不要一味追求“AI 模型更复杂”，而要优先补“规则解释、日志记录、人工确认、测试闭环”。

### 3.4 任何功能上线前都要做 3 类测试

1. 专项测试：验证这个新功能本身是否生效。
2. 回归测试：验证旧功能没有被改坏。
3. 异常测试：验证边界场景、误触发、断网、误识别时系统如何表现。

### 3.5 当前项目的高风险点不要乱动

以下位置能不动就不动，必须改时先单独验证：

- AI 颜色转换链路。
- `fb_lock` 的 LCD 互斥机制。
- 开发板 AI 进程启停逻辑。
- `server/app.py` 中 ffmpeg 子进程逻辑。
- 现有库存/事件表的基本字段语义。

## 4. 推荐开发阶段总览

建议按 7 个阶段推进。前 5 个是主体功能，后 2 个是比赛交付与加分能力。

| 阶段 | 功能主题 | 目标 | 优先级 |
| --- | --- | --- | --- |
| 阶段 0 | 基线固化与配置收敛 | 让项目变成可持续开发状态 | 必做 |
| 阶段 1 | 真实事件识别增强 | 减少误判，识别真实进出事件 | 必做 |
| 阶段 2 | 分层识别与粗分类 | 满足赛题“至少两级识别能力” | 必做 |
| 阶段 3 | 库存模型升级与部分取出 | 支持数量/状态更新与部分取出 | 必做 |
| 阶段 4 | 闭环交互与人工修正 | 让系统可确认、可修正、可解释 | 必做 |
| 阶段 5 | 包装标签/OCR/到期提醒 | 冲击加分项和展示效果 | 选做但强烈建议 |
| 阶段 6 | 测试评估、文档、演示材料 | 满足答辩和提交要求 | 必做 |

你后续如果让我带着继续做，建议按这个顺序一阶段一阶段实现，而不是同时开很多功能。

## 5. 阶段 0：基线固化与配置收敛

### 5.1 目标

这一阶段不是给评委看的功能，而是给你后面持续开发打地基。没有它，后面每加一个功能都容易把系统改坏。

### 5.2 与前面功能的兼容关系

这是所有后续功能的底座，不做这个阶段，后面所有功能都可能重复改硬编码地址、重复加日志、重复猜当前运行文件在哪里。

### 5.3 实践构成

建议涉及的内容：

- 把 IP、端口、RTSP 地址、模型路径、同步间隔做成统一配置。
- 补一份“当前实际部署版本说明”。
- 统一日志输出格式。
- 给库存数据库做备份导出方式。
- 建立简单测试样例目录和测试记录表。

建议涉及的文件：

- `deploy/fridge_mgr.py`
- `server/app.py`
- `README.md`
- `docs/AI二次开发文档.md`
- 新增一个配置文件，例如 `deploy/runtime_config.json` 或 `server/config.py`

### 5.4 详细实现流程

1. 先梳理当前所有硬编码配置。
2. 设计一个统一配置结构，不要一开始拆太复杂。
3. 先让 `deploy/fridge_mgr.py` 和 `server/app.py` 都从统一配置读取。
4. 保留默认值，避免因为配置缺失导致程序起不来。
5. 给启动日志增加关键配置信息打印。
6. 增加数据库导出与备份说明。
7. 写一个最基础的“回归检查清单”。

## 6. 阶段 1：真实事件识别增强

### 6.1 目标

赛题最核心的 20 分之一就是这个：系统必须区分“真的放入/取出物体”和“只是手伸进去晃了一下、挡住了镜头、整理了一下但没有变化”。

你当前项目已经有“开门前后对比”的基础，这很好，因为它天然符合赛题鼓励的“先做真实事件判断，再做细识别”的思路。但还需要更稳。

### 6.2 与前面功能的兼容关系

这一阶段是后面所有功能的输入源。

- 如果事件识别不稳，库存一定乱。
- 如果事件识别不稳，后面的部分取出和人工确认都会很难做。
- 所以后面所有功能都必须建立在“更稳定的事件识别结果”之上。

### 6.3 实践构成

建议新增或增强的能力：

- 门开期间记录多个时刻的检测结果，而不是只记一次。
- 关门前做稳定性确认，避免只凭最后一帧判断。
- 区分“有人手出现”与“库存有变化”。
- 事件结果带上置信度、变化原因、前后对比摘要。
- 事件状态支持 `auto_confirmed`、`needs_review`。

建议主要改动位置：

- `deploy/fridge_mgr.py`
- `server/app.py`
- `server/templates/index.html`
- 如需要，可在数据库或 JSON 结构中增加事件字段

### 6.4 详细实现流程

1. 明确当前事件判断逻辑的输入和输出。
2. 增加门开期间的检测缓存，例如 baseline、mid snapshots、after。
3. 设计稳定事件判定规则，例如：
   - 至少连续 N 次检测到相同差异才判定有效。
   - 如果只检测到 `person` 或只是框数量抖动，不触发库存更新。
   - 如果变化幅度太小或前后不一致，标记为 `needs_review`。
4. 给每条事件增加结构化字段：
   - `event_source`
   - `before_snapshot`
   - `after_snapshot`
   - `confidence`
   - `review_status`
5. Web 页面增加“自动确认/待确认”显示。
6. 保持旧事件数据兼容，不要让旧数据读不出来。

## 7. 阶段 2：分层识别与粗分类能力

### 7.1 目标

赛题明确要求至少做两级：

1. 判断有没有真实新增/减少物体。
2. 对这些物体做粗粒度分类，例如蔬果类、肉蛋生鲜类、饮料乳品类、包装食品类。

你现在已经有具体食材名称映射，但缺少一个专门的“粗分类层”。比赛里这层很重要，因为它能让系统在识别不够准时仍然保底工作。

### 7.2 与前面功能的兼容关系

- 阶段 2 必须建立在阶段 1 的稳定事件识别之上。
- 后面的库存更新、部分取出、提醒，都应该既能处理“具体名称”，也能处理“只有粗分类”的情况。

### 7.3 实践构成

建议实现：

- 建立 `fine_class -> coarse_category` 映射表。
- 当具体识别不稳定时，允许只落粗分类。
- 库存项增加：`category`、`name_confidence`、`category_confidence`。
- Web/LCD 至少显示粗分类。

建议改动文件：

- `deploy/fridge_mgr.py`
- `server/app.py`
- `server/templates/index.html`
- 如有需要，补一个分类映射配置文件

### 7.4 详细实现流程

1. 先列出当前已识别到的食材名称清单。
2. 建立粗分类映射，例如：
   - 苹果、香蕉、橙子 -> 蔬果类
   - 鸡蛋、牛奶 -> 肉蛋乳品类
   - 瓶装饮品 -> 饮料类
   - 披萨、蛋糕、面包 -> 包装食品/即食食品类
3. 改造事件和库存结构，让它支持“具体名称缺失但粗分类存在”。
4. 前端展示时优先显示具体名称，没有具体名称时显示粗分类。
5. 所有旧数据仍然能显示。

## 8. 阶段 3：库存模型升级与部分取出处理

### 8.1 目标

### 8.2 与前面功能的兼容关系

### 8.3 实践构成

- `item_type`: discrete / packaged / bulk
- `count`
- `unit`
- `state_level`: 100 / 75 / 50 / 25 / empty
- `last_event_type`
- `review_status`

推荐策略：

- 离散物体：苹果、鸡蛋、饮料瓶，用件数。
- 包装物体：盒装、袋装食品，用“剩余等级”。
- 不好精确判断的物体：进入“待确认修正”。

### 8.4 详细实现流程

1. 先定义库存数据模型。
2. 给现有物体划分类型：离散类、包装类、难判断类。
3. 对离散类直接按数量增减。
4. 对包装类做离散等级估计，例如 `full`、`half`、`low`。
5. 如果系统无法可靠判断部分取出，则把事件标为待确认，并提示用户手动修正。
6. Web 页面增加手动调整入口。
7. 保持原有旧库存数据兼容。

## 10. 阶段 5：包装标签、OCR、到期提醒

### 10.1 目标

这是非常适合拿来做比赛加分和演示亮点的功能。

赛题明确鼓励你探索：

- 对袋装、盒装食品读取包装标签信息。
- 用标签里的名称、类别、日期辅助建档。
- 做库存建档、到期提醒。

### 10.2 与前面功能的兼容关系

- 这一步不是单独的新系统，而是建立在阶段 3 的库存模型之上。
- OCR/标签识别出来的数据，要能填回现有库存结构，而不是另存一份孤立信息。

### 10.3 实践构成

建议实现：

- 新包装物进入时，抓取一张候选图片。
- 在服务端做 OCR 或者先接入可替代的云端识别。
- 提取：名称、类别、生产日期/保质期/到期日。
- 如果 OCR 不稳定，允许用户手动补录。
- 在 Web 页面增加“即将到期”提示。

建议主要改动：

- `server/app.py`
- `server/templates/index.html`
- 库存数据结构
- 如有需要，新建一个 OCR 辅助模块

### 10.4 详细实现流程

1. 先不要追求板端 OCR，优先在服务端做。
2. 定义包装类库存的扩展字段：
   - `label_name`
   - `label_category`
   - `production_date`
   - `expiry_date`
   - `ocr_confidence`
3. 新包装入库时，允许触发 OCR。
4. OCR 成功就自动填充；失败就提示手动补录。
5. 到期提醒逻辑先做简单规则版，不要一开始做太复杂。

## 14. 你后面实际使用 AI 时的建议

为了让 AI 更稳定地帮你开发，这里有几个实用建议：

1. 每次只让 AI 做一个阶段或一个子功能，不要一次塞 5 个目标。
2. 每次 Prompt 都要明确：
   - 项目路径
   - 当前主运行文件
   - 目标功能
   - 兼容要求
   - 不要乱动的高风险位置
   - 修改完成后要给测试步骤
3. 每次让 AI 改完以后，都让它做：
   - 改动说明
